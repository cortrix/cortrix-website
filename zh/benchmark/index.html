为什么选择这五个 BEIR 子集？

这组 benchmark 选择五个不同检索方向，让结果更像整体检索质量信号，而不是单一领域的最佳样例。

选择逻辑

我们想表达的信息很简单：Cortrix 应该能在不同 query、corpus 和 relevance 结构下保持检索质量。

这五个子集覆盖 scientific claims、finance QA、大规模 multi-hop QA、biomedical relevance 和 argument retrieval。组合起来，它们会同时考察小型精确语料、大 corpus 压力、密集相关性标签和复杂排序意图。

这样页面不需要让访客相信某一个好看的单点结果。它表达的是：Cortrix 正在用公开 BEIR 输入和同一组 top-ten 质量指标，覆盖不同检索场景做压力验证。

用一句话理解 BEIR

这是检索 benchmark，不是端到端 agent demo

BEIR-style evaluation 关注一个窄问题：给定 query 和 corpus，系统能不能把相关文档放到靠前位置。因此页面主指标是 Recall@10 和 NDCG@10。

如何读指标

Recall 看是否找得到，NDCG 看是否排得靠前。

Recall@10 衡量相关材料是否出现在前十个结果中；NDCG@10 还会奖励相关材料在 top-ten list 中排得更靠前。

范围边界

只声明 retrieval quality

本 benchmark 不用于宣称第三方产品优势、业务结果提升、memory 质量、audit 覆盖或端到端 RAG 回答质量。

可复现包

  • Dataset manifest：corpus、queries、qrels、loader version 和 source snapshot。
  • Run config：profile、top_k、retrieval settings、run count 和 environment notes。
  • Metrics artifact：Recall@10、NDCG@10、latency summary、raw results 和 failure counts。

正式发布前还要补什么

公开 release package 应该附上 raw JSON、CSV summary、methodology notes、reproducibility notes 和 artifact links，让访客可以检查每个数字如何产生。
