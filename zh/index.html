Agent 数据底座

AI Agent 需要的不只是一次检索调用，还需要能承接检索、记忆、可追踪性和工作流上下文的语义存储层。

Agent-Native

围绕 Agent 数据工作流设计，把语义处理、记忆和可追踪性当作共享存储问题，而不是散落在各处的对接脚本。

语义存储

文档经过解析、切分、向量化和索引后进入共享语义层，让工作流可以围绕含义和来源上下文查询。

适配现有系统

Cortrix 的目标是和现有数据库、工具和 Agent 工作流共存，为检索、记忆和可追踪性提供统一评估入口。

一个引擎，Agent 数据基础层

Cortrix 以 v1.0 发布目标组织语义摄取、混合检索与 reranker、交互记忆和来源上下文，并为 REST、MCP 和 SDK 工作流预留清晰边界。当前发布能力和后续路线图分开呈现，方便开发者知道现在能采用什么、后续还能期待什么。

Core

语义处理链路

面向 PDF、Word、Markdown 等常见文档的 v1.0 SPC 目标路径：Docling 解析、OCR fallback、parent-child chunking、NER 与摘要增强、embedding 和索引，。

Core

混合检索 + Reranker

P-HNSW 向量相似度、FTS5 + BM25 关键词检索可通过 RRF 融合，再用 bge-reranker-v2-m3 做精排，作为 v1.0 发布目标的检索路径。

Core

AI 交互记忆

持久化会话记忆属于 v1.0 目标范围：基于 LLM 的事实提取、fact / preference / event 类型记忆、衰减机制、用户隔离、透明 API 和可检查的审计上下文。

Core

Agent 可观测性

session、trace 和 agent headers 属于 v1.0 目标路径；检索反馈学习保留为后续 roadmap，本页不把自动自学习排序写成当前能力。

Core

来源级可追踪性

Cortrix 目标是让答案能够回到贡献过的 chunk、文档和对话轮次，便于调试、review 和建立信任；帮助团队检查答案背后的来源。

Core

Namespace 与跨域查询

按项目、团队或租户隔离数据；跨 namespace 检索作为 v1.0 目标路径保留统一排序语义，。

Core

检索质量基础

parent-child chunking 属于发布目标质量基础；contextual retrieval、multi-query patterns、retrieval grading 和 hypothetical-question indexing 在公开范围冻结前作为评估路径处理。

Integration

pgCortrix Extension

pgCortrix 是把语义存储能力靠近现有 Postgres 基础设施的 extension 路径；是否需要独立服务和具体部署拓扑，需要按 release 范围逐项验证。

Integration

MCP Server

Model Context Protocol server 支持按 published scope 处理，用于把 Cortrix 能力暴露给已配置使用 MCP 的 AI 工具和本地工作流。

Local

本地 Embedding 路径（bge-m3）

通过 ONNX Runtime 集成 bge-m3 多语言 embedding 模型是 v1.0 目标中的本地 embedding 路径；是否需要外部服务取决于部署模型与 launch verification。

路径

连接器路径

面向 MCP、CLI、REST、HTTP upload、directory watch、DB import 和自定义数据源路径设计；公开集成会围绕开发者常见的本地工作流逐步完善。

Interface

Python SDK + REST + Web UI

pip install cortrix 是目标 Python client 路径；HTTP API 和本地 dashboard 路径用于文档管理、搜索和 AI chat，覆盖文档管理、搜索和 AI chat。

架构

从文档摄取到混合检索、reranker、记忆和来源上下文，Cortrix 提供一个面向 Agent 数据工作流的 v1.0 目标系统视图。

该图是系统概览，不代表 benchmark 结果、生产规模保证或当前已公开的托管/商业产品。

Docker 或 PG Extension 路径

可以评估 docker-compose、本地 standalone server 或 pgcortrix PostgreSQL extension 路径，最终按 published scope 和团队技术栈确认。

用于 review 的反馈信号

可观测性和反馈信号帮助团队检查哪些内容支持了 Agent 答案；基于反馈信号的自动学习保留为 roadmap，后续再进入公开路线图。

分范围的连接器路径

设计上面向 MCP、CLI workflow、HTTP upload、filesystem watcher 和 framework adapter；每个公开连接器随验证成熟度推进。

评估本地路径

以下命令展示 v1.0 目标中的本地流程；镜像、包和 API 可用性需要在 public documentation 阶段确认。

1

拉取并运行

Terminal
docker pull cortrix/cortrix:latest
docker run -d -p 8080:8080 --name cortrix cortrix/cortrix:latest
2

上传文档

Terminal
curl -X POST http://localhost:8080/api/v1/documents/upload \
  -F "file=@your-document.pdf" \
  -F "namespace=default"
3

语义查询

Terminal
curl -X POST http://localhost:8080/api/v1/query \
  -H "Content-Type: application/json" \
  -d '{"query": "How does authentication work?", "namespace": "default"}'

也可以使用 Python SDK：

Python
pip install cortrix

from cortrix import Client
client = Client("http://localhost:8080")
client.documents.upload("your-document.pdf", namespace="default")
results = client.query("How does authentication work?", namespace="default")

面向真实 Agent 数据工作流

Cortrix 聚焦 Agent 工作流背后的共享数据层：文档、检索上下文、记忆、反馈信号和可追踪性。

Agent 工作流存储

为需要持续语义记忆和可检查来源上下文的工作流提供 v1.0 目标评估基础。

可 review 的 Agent 上下文

帮助团队查看哪些文档、chunk 和对话轮次参与了答案或工作流步骤。

工作流编排

作为 MCP、CLI、REST 和框架型工作流可评估的共享语义层，具体范围按 release 状态区分。

开发者知识工作流

围绕文档、工作流上下文和 Agent 记忆建立语义层，同时保留现有数据库和工具。

能力边界对照

这不是攻击数据库或框架，而是说明 Cortrix 如何减少胶水代码拼接的数据平台现状。

能力Cortrix 范围常见拼装方式
文档摄取v1.0 目标SPC 摄取基础常由 parser、chunker 和索引脚本分开维护
Embeddingv1.0 目标本地 embedding 路径（bge-m3, ONNX）常由外部模型或服务处理
向量检索v1.0 目标P-HNSW in-process 路径常交给独立 vector database
关键词检索v1.0 目标FTS5 + BM25 路径可能需要独立 search component
混合融合v1.0 目标RRF fusion 路径常需要自定义编排代码
Rerankerv1.0 目标bge-reranker-v2-m3 路径可能需要外部服务或自定义步骤
跨 Namespace 查询v1.0 目标scatter-gather + 统一排序路径常由客户端编排处理
Advanced RAG已规划 / 待验证按 release 范围验证检索质量模式常由框架插件或自定义管线处理
AI 记忆v1.0 目标typed memory 与提取路径常接入独立 memory service 或自建 store
来源级可追踪性v1.0 目标内容级引用和归因上下文不同栈的 tracing 范围差异较大

检索反馈学习Roadmap基于反馈信号的自动学习通常需要自定义 feedback loop
PostgreSQL 集成已规划 / 待验证pgCortrix extension 路径常作为独立服务运行
Workflow Connectors已规划 / 待验证MCP、CLI、REST 和框架适配路径通常由框架或项目自行适配
MCP Serverv1.0 目标MCP server 路径可用性取决于所选技术栈
范围边界v1.0 目标v1.0 目标 / 已规划 / 被阻塞 / 愿景级常分散在多个文档或 issue 中
部署Docker / server / PG extension 路径按团队技术栈选择通常需要多个服务
