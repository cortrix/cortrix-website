Agent 数据底座

AI Agent 需要的不只是一次检索调用，还需要能承接检索、记忆、可追踪性和工作流上下文的语义存储层。

Agent-Native

围绕 Agent 数据工作流设计，把语义处理、记忆和可追踪性当作共享存储问题，而不是散落在各处的对接脚本。

语义存储

文档经过解析、切分、向量化和索引后进入共享语义层，让工作流可以围绕含义和来源上下文查询。

适配现有系统

Cortrix 的目标是和现有数据库、工具和 Agent 工作流共存，为检索、记忆和可追踪性提供统一评估入口。

一个引擎，Agent 数据底座

Cortrix 将语义摄取、混合检索与 reranker、交互记忆和因果链上下文收拢到共享引擎基础中，并提供 REST、MCP 和 SDK 接口。

Core

语义处理链路

面向 PDF、Word、Markdown 等文档的自动摄取路径：Docling 解析、OCR fallback、parent-child chunking、NER 与摘要增强、embedding 和索引一步完成。

Core

混合检索 + Reranker

P-HNSW 向量相似度、FTS5 + BM25 关键词检索通过 RRF 融合，再用 bge-reranker-v2-m3 做精排，服务更高质量的检索工作流。

Core

AI 交互记忆

持久化会话记忆、基于 LLM 的事实提取、fact / preference / event 类型记忆、衰减机制、用户隔离、透明 API 和可检查的审计上下文。

Core

Agent 可观测性与自学习

session、trace 和 agent headers 端到端流转；检索反馈记录真正有帮助的 chunk，并用 useful_ratio 静默影响后续排序，不需要额外训练管线。

Core

内容级可追踪性

答案可以回到贡献过的 chunk、文档和对话轮次，便于调试、review 和建立信任。

Core

Namespace 与跨域查询

按项目、团队或租户隔离数据，并为跨 namespace 检索保留统一排序路径。

Core

检索质量基础

parent-child chunking、contextual retrieval、multi-query patterns、retrieval grading 和 hypothetical-question indexing 作为存储层评估路径处理。

Integration

pgCortrix Extension

将语义存储能力作为 PostgreSQL extension 带入现有 Postgres 基础设施；该路径按公开 release 范围验证。

Integration

MCP Server

Model Context Protocol server 路径把 Cortrix 能力暴露给已配置使用 MCP 的 AI 工具和本地工作流。

Local

本地 Embedding 路径（bge-m3）

通过 ONNX Runtime 集成 bge-m3 多语言 embedding 模型，不需要外部 embedding 服务。

Ecosystem

连接器生态

面向 MCP、CLI、REST、HTTP upload、directory watch、DB import 和自定义数据源路径设计，随公开集成成熟逐步扩展。

Interface

Python SDK + REST + Web UI

pip install cortrix 提供类型安全的 Python client；HTTP API 面向任意语言，本地 dashboard 路径用于文档管理、搜索和 AI chat。

架构

从文档摄取到混合检索、reranker、记忆和内容级因果链上下文，Cortrix 提供一个面向 Agent 数据工作流的系统视图。

该图是系统概览，不代表 benchmark 结果、生产规模保证或当前已公开的托管/商业产品。

Docker 或 PG Extension

可以通过 docker-compose 一条命令、本地 standalone server，或 pgcortrix PostgreSQL extension 路径部署，按团队现有技术栈选择。

从每次查询中学习

检索反馈闭环会记录真正帮助 Agent 的 chunk，每个 chunk 累积 useful_ratio，在后续结果中逐步提升相关性，不需要 fine-tuning 或额外管线。

通用连接器方向

设计上面向 MCP、CLI workflow、HTTP upload、filesystem watcher 和 framework adapter；具体连接器随公开集成成熟度推进。

快速开始

用本地路径开始评估语义检索，不等同于生产环境承诺。

1

拉取并运行

Terminal
docker pull cortrix/cortrix:latest
docker run -d -p 8080:8080 --name cortrix cortrix/cortrix:latest
2

上传文档

Terminal
curl -X POST http://localhost:8080/api/v1/documents/upload \
  -F "file=@your-document.pdf" \
  -F "namespace=default"
3

语义查询

Terminal
curl -X POST http://localhost:8080/api/v1/query \
  -H "Content-Type: application/json" \
  -d '{"query": "How does authentication work?", "namespace": "default"}'

也可以使用 Python SDK：

Python
pip install cortrix

from cortrix import Client
client = Client("http://localhost:8080")
client.documents.upload("your-document.pdf", namespace="default")
results = client.query("How does authentication work?", namespace="default")

面向真实 Agent 数据工作流

Cortrix 聚焦 Agent 工作流背后的共享数据层：文档、检索上下文、记忆、反馈和可追踪性。

Agent 工作流存储

为需要持续语义记忆和可检查决策上下文的工作流提供评估基础。

可 review 的 Agent 上下文

帮助团队查看哪些文档、chunk 和对话轮次参与了答案或工作流步骤。

工作流编排

作为 MCP、CLI、REST 和框架型工作流可读写的共享语义层，具体范围按实现状态区分。

开发者知识工作流

围绕文档、工作流上下文和 Agent 记忆建立语义层，同时保留现有数据库和工具。

能力边界对照

这不是攻击数据库或框架，而是说明 Cortrix 如何减少胶水代码拼接的数据平台现状。

能力Cortrix 范围常见拼装方式
文档摄取当前开源SPC 摄取基础常由 parser、chunker 和索引脚本分开维护
Embedding当前开源本地 embedding 路径（bge-m3, ONNX）常由外部模型或服务处理
向量检索当前开源P-HNSW in-process 路径常交给独立 vector database
关键词检索当前开源FTS5 + BM25 路径可能需要独立 search component
混合融合当前开源RRF fusion 路径常需要自定义编排代码
Reranker当前开源bge-reranker-v2-m3 路径可能需要外部服务或自定义步骤
跨 Namespace 查询当前开源scatter-gather + 统一排序路径常由客户端编排处理
Advanced RAG已规划 / 待验证按 release 范围验证检索质量模式常由框架插件或自定义管线处理
AI 记忆当前开源typed memory 与提取路径常接入独立 memory service 或自建 store
因果链上下文当前开源内容级引用和归因上下文不同栈的 tracing 范围差异较大
Agent 可观测性当前开源session、trace 和 feedback signals常需要独立 instrumentation
自学习检索当前开源useful_ratio feedback 路径通常需要自定义 feedback loop
PostgreSQL 集成已规划 / 待验证pgCortrix extension 路径常作为独立服务运行
Workflow Connectors已规划 / 待验证MCP、CLI、REST 和框架适配路径通常由框架或项目自行适配
MCP Server当前开源MCP server 路径可用性取决于所选技术栈
范围边界当前开源当前开源 / 已规划 / 被阻塞 / 愿景级常分散在多个文档或 issue 中
部署Single binary / Docker / PG ext通常需要多个服务
